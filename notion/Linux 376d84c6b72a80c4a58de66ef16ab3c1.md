```wasm
Linux Permissões & Logs — Resumo

  ---
  chmod — Permissões

  chmod 775 /pasta

  Dono  → 7 → rwx (lê, escreve, executa)
  Grupo → 7 → rwx (lê, escreve, executa)
  Outro → 5 → r-x (lê e executa, NÃO escreve)

  r = 4 | w = 2 | x = 1  →  soma os que quer dar

  Valores comuns:

  ┌────────┬───────────┬───────────────────────────────┐
  │ Número │ Permissão │          Uso típico           │
  ├────────┼───────────┼───────────────────────────────┤
  │ 777    │ rwxrwxrwx │ Nunca em produção             │
  ├────────┼───────────┼───────────────────────────────┤
  │ 775    │ rwxrwxr-x │ Pastas compartilhadas         │
  ├────────┼───────────┼───────────────────────────────┤
  │ 755    │ rwxr-xr-x │ Pastas públicas (mais seguro) │
  ├────────┼───────────┼───────────────────────────────┤
  │ 644    │ rw-r--r-- │ Arquivos normais              │
  └────────┴───────────┴───────────────────────────────┘

  ---
  Usuários & Grupos

  Usuário = pessoa individual (ec2-user, bruno, root)
  Grupo   = conjunto de usuários (developers, www-data)
  Arquivo = tem um dono + um grupo atribuído

  su bruno     → troca de usuário
  whoami       → mostra quem você é
  groups       → mostra seus grupos
  ls -l        → lista arquivos com dono, grupo e permissões

  ---
  Lendo um log de erro

  [data hora] [módulo:nível] [pid] [client IP:porta] mensagem

  Níveis:
  error → grave, algo quebrou
  warn  → aviso, algo pode quebrar
  info  → informativo

  Sequência de troubleshooting:
  1. Nível → error ou warn?
  2. Módulo → onde está o problema (proxy_fcgi = PHP)
  3. Mensagem → o que aconteceu (Permission denied)
  4. Arquivo → qual caminho está com problema

  ---
  Caso real — Bitnami WordPress

  Erro:    Permission denied → wp-includes/version.php
  Causa:   PHP (daemon) sem permissão de leitura na pasta
  Solução: sudo chmod 755 /opt/bitnami/wordpress/wp-includes
  Log:     sudo tail /opt/bitnami/apache/logs/error_log

```

```wasm
  su = Switch User

  ---
  Agora o que é grupo

  Grupo é uma lista VIP. Você coloca vários usuários numa lista e dá permissão pra lista inteira de uma vez.

  Grupo "developers"
  ├── ec2-user
  ├── bruno
  └── maria

  Grupo "sudo"        ← quem pode virar root
  ├── ec2-user
  └── (bruno NÃO está aqui)

  Se um arquivo pertence ao grupo developers, ec2-user, bruno e maria conseguem acessar — sem precisar dar permissão um
  por um.

  ---
  Ver os grupos do seu usuário

  groups              # lista os grupos do usuário atual
  id                  # mostra usuário + todos os grupos com número

  Numa EC2 Amazon Linux, o ec2-user já vem no grupo sudo (ou wheel) por padrão — é por isso que você consegue rodar sudo
  com ele.

  ---
  Exemplo prático real

  Arquivo:  /var/www/index.html
  Dono:     ec2-user
  Grupo:    www-data
  Permissão: 664

  → ec2-user pode ler e escrever
  → nginx (roda como www-data) pode ler e escrever
  → bruno NÃO consegue escrever (é "outro", só lê)

  Se você quiser que bruno também escreva, adiciona ele ao grupo www-data:

  sudo usermod -aG www-data bruno

  ---
  Resumo mental

  Usuário = pessoa individual
  Grupo   = time/categoria de pessoas
  Arquivo = tem um dono (pessoa) + um grupo (time)
```
