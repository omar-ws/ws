## Fluxo de uma aplicacao real:

Fluxo completo:

Usuário
↓
ALB Externo (subnet pública)
↓
EC2 Web - frontend (subnet privada) (sempre um S3 pois fica sobrecarregando a instancia)
↓ (quando precisa processar/salvar)
ALB Interno (subnet privada)
↓
EC2 App - backend (subnet privada) (sempre na subrede privada)
↓
RDS (subnet privada)

caminho

user ⇒ s3 ⇒ fecth para o elb ⇒ ec2 app ⇒ rds

no caso de um web e app

user ⇒ cloudfront ⇒ s3 ⇒ fecth elb  ⇒ elb (pub) ⇒ ec2-app (priv sempre) ⇒ RDS

SG: “quem pode se conectar?” EX: EC2 acessa porta 3306 RDS

IAM: “quem pode usar?” EX: ec2 pode ler S3

exemplo de quando usar cada um:

EC2 App → RDS          = SG (conexão de rede, porta 3306)
EC2 App → S3           = IAM Role (ação: s3:GetObject)
EC2 App → Secrets Manager = IAM Role (ação: secretsmanager:GetSecretValue)
EC2 Web → EC2 App      = SG (conexão de rede)
Lambda  → DynamoDB     = IAM Role (ação: dynamodb:PutItem)

Snapshot e apenas a foto do disco da EBS

AMI e a foto da maquina completa, desde os snapshots ate o SO 

## Logica ELB:

  Internet → ALB (Listener porta 443) → Rule → Target Group → EC2 App

user⇒internet⇒elb⇒listen⇒destino

logica ALB para regras de rota
ALB recebe requisição
→ /login*    → Target Group Auth
→ /checkout* → Target Group Payment
→ /produtos* → Target Group Catalog
